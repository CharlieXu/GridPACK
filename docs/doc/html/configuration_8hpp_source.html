1 // We expect execution of a GridPACK application to be determined in part
2 // by a single input configuration file (perhaps extracted from a database
3 // or web service). We assume that a configuration file is a binding of a
4 // hierarchically structured set of keys to a set of values. The
5 // Configuration module is used to access values and abstract the structure
6 // of the configuration file which might be concretely represented in a ways
7 // including XML or with a custom parser. Different components within
8 // GridPACK will share a common configuration file and we expect then the
9 // top-level of the key name space to identify the module that is primarily
10 // associated with a configuration parameter. This provides extensibility over
11 // time and insulates modules from each other’s internals.
12 
13 // Keys are represented by std::string with the character
14 // Configuration::KeySeparator,‘.’ (properly escaped in literals) used as
15 // a name specifier. So “foo.bar” is a key associated by convention with
16 // module “bar”.
17 
18 // Values are typed and may be of the following primitive types: bool, int, double
19 // or std::string. We also allow a value to be a std::vector<double>.
20 
21 // We use the term cursor to a common prefix on a set of names. Access to a value
22 // may be specified by a complete key or by a cursor and the corresponding suffix
23 // relative to that cursor. Thus a cursor may represent the prefix “foo” and
24 // relative to that cursor the string “bar” selects the same values as
25 // “foo.bar”. Aside from factoring out common information for the client,
26 // it allows a common configuration substructure do be used by different modules.
